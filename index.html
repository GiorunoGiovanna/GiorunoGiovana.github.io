在前端工程中，脚手架是一个常用的工具，它可以帮助开发者快速搭建一个可以运行的，定制化的前端工程项目。常用脚手架有vue-cli,vite-cli等。

脚手架有以下作用：
  • 自由配置需要的组件/依赖
  • 可以统一规定模板，使用统一的规则例如代码规范eslint，代码提交规范husky等
  • 可以规定使用依赖的版本，防止出现一些兼容性错误

以vue-cli为例

vue-cli是vue官方推荐使用的vue脚手架。使用它可以轻松配置一个vue项目。vue-cli使用步骤如下：

  • 全局安装vue-cli，npm i vue/cli -g
  • 在命令行界面使用vue create demo这个命令创建一个名为demo的项目
  • 后续会有交互式的命令行提问，供用户设置项目的内容，包括是否使用vuex，是否使用typescript等
  • 自动安装依赖
  • 完成项目

因此，我们所制作的脚手架也要有这几个功能。

Github Actions是GitHub推出的一个CI/CD工具，可以在这个自动化部署的流程中进行代码校验，代码测试，代码部署等动作，使得开发者只需要关心代码本身，而不必考虑部署问题。一般而言，流程如下：

React-hook

在React中由两种组件写法，一种是类式组件，另一种是函数式组件，其中函数式组件是无状态组件，即没有生命周期，也没有数据状态管理state等。因此，为了让函数式组件也可以像类组件一样具有状态，所以使用React-hook。

为什么使用函数式组件+hook

  1. 函数式组件+hook写的代码逻辑性更强，可以抽离公共的方法，公共组件。
  2. 写起来更有函数即组件，无疑也提高代码的开发效率、
  3. react-hooks可能把庞大的class组件，化整为零成很多小组件，useMemo等方法让组件或者变量制定一个适合自己的独立的渲染空间，一定程度上可以提高性能，减少渲染次数。

如何使用hook

  1. useState 数据存储，派发更新

    useState返回一个数组，第一项是state此时的值，第二项是state的派发更新函数，即想要修改state就需要通过这个更新函数去修改，值得注意的是，state发生更改会导致整个函数组件从头到尾执行一次，因此需要配合useMemo，useCallback等api使用，否则回拖垮性能

  2. useEffect 组件更新副作用钩子

    useEffect是一个在组件发生更新后触发的钩子，其中包括初次绘制，state更新，props更新。因此想要更好的使用这个钩子应当适当添加监听对象，即这个钩子的第二个参数，这是一个数组，当数组内的状态发生改变的时候，才触发这个钩子。

  3. useLayoutEffect 渲染更新之前的 useEffect

    类似于useEffect,useEffect 执行顺序 组件更新挂载完成 -> 浏览器dom 绘制完成 -> 执行useEffect回调 。

    useLayoutEffect 执行顺序 组件更新挂载完成 -> 执行useLayoutEffect回调-> 浏览器dom 绘制完成

  4. useRef 获取元素，缓存数据

    和传统的class组件ref一样，react-hooks 也提供获取元素方法 useRef,它有一个参数可以作为缓存数据的初始值，返回值可以被dom元素ref标记，可以获取被标记的元素节点.

  5. useContext 获取来自祖先组件的context值
  6. useMemo 减少不必要的渲染

    使用useMemo可以让在memo中的方法仅在它规定的监听对象发生更改的时候才重新执行

  7. useCallback useMemo版本的回调函数

    跟useMemo很像，但是useMemo返回的是函数的运行结果，而useCallback返回函数。

Vue中keep-alive缓存组件

Vue中的keep-alive是用于保持组件存活的一个功能，因为在Vue中，当组件被切换成另一个的时候，这个旧组件会被销毁，如果不进行缓存，再次打开这个组件的时候就需要重复渲染。因此，对于需要频繁切换显示的组件可以使用keep-alive缓存组件

使用方法

keep-alive类似于router-view是一个抽象组件，它本身不会渲染dom，只会保存它标签下的组件。

  • <keep-alive>标签特有的属性include以及exclude字段来进行，include表示需要缓存的页面，exclude表示不需要缓存的页面，可以看见我在data中注册了黑名单和白名单，切记是写上组件中注册的name属性，而不是路由中写的name属性。值得注意的是，当你同一个name的组件，既写在exclude，又写在include中，脚踏两只船时，exclude的权级比include要高，例如a组件脚踏两只船的时候，a组件是不会被缓存。
  • 通过设置max，可以控制可缓存组件的最大数，当缓存的数量达到最大时，采用LRU算法，将最近最久未使用的页面予以淘汰，再将新页面进行缓存

浏览器iframe初探

iframe是html5的一个标签，专门用来内嵌一个网页到dom中的，由于网站之间是相对独立的，因此iframe是由许多规则的。iframe是嵌入式框架, 是html标签, 还是一个内联元素， iframe 元素会创建包含另外一个文档的内联框架（即行内框架) ，说白了iframe用来在页面嵌入其他页面。

iframe常见属性

  • frameborder是否显示边框，1(yes),0(no)
  • height/width
  • name框架的名称，window.frames[name]时专用的属性。
  • scrolling框架的是否滚动。yes,no,auto
  • src内框架的地址，可以使页面地址，也可以是图片的地址
  • sandbox沙箱限制
    • allow-same-origin允许将内容作为普通来源对待。
    • allow-top-navigation嵌入的页面的上下文可以导航（加载）内容到顶级的浏览上下文环境（browsing context）。
    • allow-forms允许表单提交。
    • allow-script 允许脚本执行。

iframe的优缺点

优点：

  • 页面和程序分离，几乎不会收到外界任何js或css影响
  • 可以通过iframe嵌套页面，提高代码复用率，比如页面的头部样式
  • 重新加载页面的时候，不需要重载iframe页内容
  • iframe可以解决第三方内容加载缓慢

缺点：

  • 复杂度高，不宜管理
  • iframe框架的内容无法被搜索引擎捕获, 所以iframe不适用于首页
  • iframe兼容性较差
  • iframe有一定的安全风险
  • iframe会阻塞主页面的Onload事件

vite与webpack比较

众所周知，vite和webpack是前端中比较流行的打包框架，它们之间各有特色，实现原理也不相同。这里探讨一下vite和webpack的区别到底有哪些，以及两者各自的优劣。

vite和webpack的区别

打包过程

  • wepack打包过程
    1. 识别入口文件
    2. 通过递归方式遍历每一个依赖。生成虚拟语法树AST
    3. 分析生成AST，编译代码，生成代码
    4. 打包
  • vite打包过程
    1. 识别入口文件
    2. 使用基于go语言的esbuild预构建依赖打包依赖文件
    3. 启动webSocket服务器
    4. 当浏览器需要资源的时候，发起get请求。vite劫持请求在服务器中整合需要的依赖进行返回

区别：

  • webpack是预先将所有依赖以及源码先全部打包，使用的是基于JavaScript语言的打包器，然后启动静态服务器。在启动过程中比较缓慢。
  • vite是将依赖通过基于go的esbuild预构建依赖，源码不做处理，等到浏览器解析到需要的JavaScript再发起请求获取资源，启动较快。

热更新过程

  • webpack热更新
    1. 在以前的webpack中，由于将所有的文件都做了预先打包，当文件发生更改的时候，就会出现某一部分的AST节点以及之后的节点可能全部失效的情况，这个时候webpack默认会重新打包一遍整个文件，开发体验较差。
    2. 现在webpack可以支持热重载（HMR），只对其中某个更改的的模块进行热更新，其他模块不需要重复打包。当然，如果项目过大，这个方法仍然体验不好。
  • vite热更新
    1. 在 Vite 中，HMR 是在原生 ESM 上执行的。当编辑一个文件时，Vite 只需要精确地使已编辑的模块与其最近的 HMR 边界之间的链失效（大多数时候只需要模块本身），使 HMR 更新始终快速，无论应用的大小。
    2. Vite 同时利用 HTTP 头来加速整个页面的重新加载（再次让浏览器为我们做更多事情）：源码模块的请求会根据 304 Not Modified 进行协商缓存，而依赖模块请求则会通过 Cache-Control,immutable 进行强缓存，因此一旦被缓存它们将不需要再次请求。
