前置概念

CI/CD

在传统项目开发中，我们需要将项目打包后将生成的文件上传到服务器来部署一个项目。而CI/CD是一种通过在应用开发阶段引入自动化来进行持续集成和部署等操作的方案。换句话说就是把手动部署任务换成了自动部署任务。

GitHub Actions

Github Actions是GitHub推出的一个CI/CD工具，可以在这个自动化部署的流程中进行代码校验，代码测试，代码部署等动作，使得开发者只需要关心代码本身，而不必考虑部署问题。一般而言，流程如下：

  1. 触发actions的动作，例如push代码
  2. actions自动执行预设好的任务
  3. 反馈

流程

  1. 创建github项目，注意生成的github page默认url为https://(你的用户名).github.io/(你的项目名)

  2. 可以自行编写/.github/workflows中actions脚本文件，例如main.yml,也可以在Actions选项卡中选择一个已经提前编写好的脚本文件

  3. 如果步骤正确，会在项目中新建一个gh-page分支，专门存放将部署好的静态文件

  4. 我们在Settings选项卡中的Pages中选择部署的分支为gh-page

  5. 此时访问url即可看到部署成功的页面了

React-hook

在React中由两种组件写法，一种是类式组件，另一种是函数式组件，其中函数式组件是无状态组件，即没有生命周期，也没有数据状态管理state等。因此，为了让函数式组件也可以像类组件一样具有状态，所以使用React-hook。

为什么使用函数式组件+hook

  1. 函数式组件+hook写的代码逻辑性更强，可以抽离公共的方法，公共组件。
  2. 写起来更有函数即组件，无疑也提高代码的开发效率、
  3. react-hooks可能把庞大的class组件，化整为零成很多小组件，useMemo等方法让组件或者变量制定一个适合自己的独立的渲染空间，一定程度上可以提高性能，减少渲染次数。

如何使用hook

  1. useState 数据存储，派发更新

    useState返回一个数组，第一项是state此时的值，第二项是state的派发更新函数，即想要修改state就需要通过这个更新函数去修改，值得注意的是，state发生更改会导致整个函数组件从头到尾执行一次，因此需要配合useMemo，useCallback等api使用，否则回拖垮性能

  2. useEffect 组件更新副作用钩子

    useEffect是一个在组件发生更新后触发的钩子，其中包括初次绘制，state更新，props更新。因此想要更好的使用这个钩子应当适当添加监听对象，即这个钩子的第二个参数，这是一个数组，当数组内的状态发生改变的时候，才触发这个钩子。

  3. useLayoutEffect 渲染更新之前的 useEffect

    类似于useEffect,useEffect 执行顺序 组件更新挂载完成 -> 浏览器dom 绘制完成 -> 执行useEffect回调 。

    useLayoutEffect 执行顺序 组件更新挂载完成 -> 执行useLayoutEffect回调-> 浏览器dom 绘制完成

  4. useRef 获取元素，缓存数据

    和传统的class组件ref一样，react-hooks 也提供获取元素方法 useRef,它有一个参数可以作为缓存数据的初始值，返回值可以被dom元素ref标记，可以获取被标记的元素节点.

  5. useContext 获取来自祖先组件的context值
  6. useMemo 减少不必要的渲染

    使用useMemo可以让在memo中的方法仅在它规定的监听对象发生更改的时候才重新执行

  7. useCallback useMemo版本的回调函数

    跟useMemo很像，但是useMemo返回的是函数的运行结果，而useCallback返回函数。

Vue中keep-alive缓存组件

Vue中的keep-alive是用于保持组件存活的一个功能，因为在Vue中，当组件被切换成另一个的时候，这个旧组件会被销毁，如果不进行缓存，再次打开这个组件的时候就需要重复渲染。因此，对于需要频繁切换显示的组件可以使用keep-alive缓存组件

使用方法

keep-alive类似于router-view是一个抽象组件，它本身不会渲染dom，只会保存它标签下的组件。

  • <keep-alive>标签特有的属性include以及exclude字段来进行，include表示需要缓存的页面，exclude表示不需要缓存的页面，可以看见我在data中注册了黑名单和白名单，切记是写上组件中注册的name属性，而不是路由中写的name属性。值得注意的是，当你同一个name的组件，既写在exclude，又写在include中，脚踏两只船时，exclude的权级比include要高，例如a组件脚踏两只船的时候，a组件是不会被缓存。
  • 通过设置max，可以控制可缓存组件的最大数，当缓存的数量达到最大时，采用LRU算法，将最近最久未使用的页面予以淘汰，再将新页面进行缓存

浏览器iframe初探

iframe是html5的一个标签，专门用来内嵌一个网页到dom中的，由于网站之间是相对独立的，因此iframe是由许多规则的。iframe是嵌入式框架, 是html标签, 还是一个内联元素， iframe 元素会创建包含另外一个文档的内联框架（即行内框架) ，说白了iframe用来在页面嵌入其他页面。

iframe常见属性

  • frameborder是否显示边框，1(yes),0(no)
  • height/width
  • name框架的名称，window.frames[name]时专用的属性。
  • scrolling框架的是否滚动。yes,no,auto
  • src内框架的地址，可以使页面地址，也可以是图片的地址
  • sandbox沙箱限制
    • allow-same-origin允许将内容作为普通来源对待。
    • allow-top-navigation嵌入的页面的上下文可以导航（加载）内容到顶级的浏览上下文环境（browsing context）。
    • allow-forms允许表单提交。
    • allow-script 允许脚本执行。

iframe的优缺点

优点：

  • 页面和程序分离，几乎不会收到外界任何js或css影响
  • 可以通过iframe嵌套页面，提高代码复用率，比如页面的头部样式
  • 重新加载页面的时候，不需要重载iframe页内容
  • iframe可以解决第三方内容加载缓慢

缺点：

  • 复杂度高，不宜管理
  • iframe框架的内容无法被搜索引擎捕获, 所以iframe不适用于首页
  • iframe兼容性较差
  • iframe有一定的安全风险
  • iframe会阻塞主页面的Onload事件

vite与webpack比较

众所周知，vite和webpack是前端中比较流行的打包框架，它们之间各有特色，实现原理也不相同。这里探讨一下vite和webpack的区别到底有哪些，以及两者各自的优劣。

vite和webpack的区别

打包过程

  • wepack打包过程
    1. 识别入口文件
    2. 通过递归方式遍历每一个依赖。生成虚拟语法树AST
    3. 分析生成AST，编译代码，生成代码
    4. 打包
  • vite打包过程
    1. 识别入口文件
    2. 使用基于go语言的esbuild预构建依赖打包依赖文件
    3. 启动webSocket服务器
    4. 当浏览器需要资源的时候，发起get请求。vite劫持请求在服务器中整合需要的依赖进行返回

区别：

  • webpack是预先将所有依赖以及源码先全部打包，使用的是基于JavaScript语言的打包器，然后启动静态服务器。在启动过程中比较缓慢。
  • vite是将依赖通过基于go的esbuild预构建依赖，源码不做处理，等到浏览器解析到需要的JavaScript再发起请求获取资源，启动较快。

热更新过程

  • webpack热更新
    1. 在以前的webpack中，由于将所有的文件都做了预先打包，当文件发生更改的时候，就会出现某一部分的AST节点以及之后的节点可能全部失效的情况，这个时候webpack默认会重新打包一遍整个文件，开发体验较差。
    2. 现在webpack可以支持热重载（HMR），只对其中某个更改的的模块进行热更新，其他模块不需要重复打包。当然，如果项目过大，这个方法仍然体验不好。
  • vite热更新
    1. 在 Vite 中，HMR 是在原生 ESM 上执行的。当编辑一个文件时，Vite 只需要精确地使已编辑的模块与其最近的 HMR 边界之间的链失效（大多数时候只需要模块本身），使 HMR 更新始终快速，无论应用的大小。
    2. Vite 同时利用 HTTP 头来加速整个页面的重新加载（再次让浏览器为我们做更多事情）：源码模块的请求会根据 304 Not Modified 进行协商缓存，而依赖模块请求则会通过 Cache-Control,immutable 进行强缓存，因此一旦被缓存它们将不需要再次请求。

http常见请求头和响应头

发表于 2023-04-03 分类于 计算机网络 Valine：

http常见请求头和响应头

请求头

  1. Accept

    Accept:text/html浏览器可以接受的服务器发回的类型

  2. Connection

    Connection:keep-alive是否保持这个tcp链接的链接状态（长连接）

  3. Host（请求目标地址）

    Host:www.baidu.com 请求报头域主要用于指定被请求资源的Internet主机和端口号，它通常从HTTP URL中提取出来的。

  4. Referer（请求来源地址）

    Referer:https://www.baidu.com/?tn=62095104_8_oem_dg 当浏览器向web服务器发送请求的时候，一般会带上Referer，告诉服务器我是从哪个页面链接过来的，服务器籍此可以获得一些信息用于处理。

  5. User-Agent

    告诉HTTP服务器， 客户端使用的操作系统和浏览器的名称和版本。

  6. Cache-control
    • Cache-Control:private 默认为private 响应只能够作为私有的缓存，不能再用户间共享
    • Cache-Control:public 响应会被缓存，并且在多用户间共享。正常情况, 如果要求HTTP认证,响应会自动设置为 private.
    • Cache-Control:no-cache 响应不会被缓存,而是实时向服务器端请求资源。
    • Cache-Control:max-age=10 设置缓存最大的有效时间，但是这个参数定义的是时间大小（比如：60）而不是确定的时间点。单位是[秒 seconds]。
    • Cache-Control:no-store 在任何条件下，响应都不会被缓存，并且不会被写入到客户端的磁盘里，这也是基于安全考虑的某些敏感的响应才会使用这个。
  7. Cookie

    Cookie是用来存储一些用户信息以便让服务器辨别用户身份的（大多数需要登录的网站上面会比较常见），比如cookie会存储一些用户的用户名和密码，当用户登录后就会在客户端产生一个cookie来存储相关信息，这样浏览器通过读取cookie的信息去服务器上验证并通过后会判定你是合法用户，从而允许查看相应网页。当然cookie里面的数据不仅仅是上述范围，还有很多信息可以存储是cookie里面，比如sessionid等。

  8. Range（用于断点续传）

    Range:bytes=0-5 指定第一个字节的位置和最后一个字节的位置。用于告诉服务器自己想取对象的哪部分。

常用的http响应头

  1. Content-Type

    Content-Type：text/html;charset=UTF-8 告诉客户端，资源文件的类型，还有字符编码，客户端通过utf-8对资源进行解码，然后对资源进行html解析。通常我们会看到有些网站是乱码的，往往就是服务器端没有返回正确的编码。

  2. Date

    Date: Tue, 03 Apr 2018 03:52:28 GMT 这个是服务端发送资源时的服务器时间，GMT是格林尼治所在地的标准时间。http协议中发送的时间都是GMT的，这主要是解决在互联网上，不同时区在相互请求资源的时候，时间混乱问题。

  3. Expires

    Expires:Sun, 1 Jan 2000 01:00:00 GMT 这个响应头也是跟缓存有关的，告诉客户端在这个时间前，可以直接访问缓存副本，很显然这个值会存在问题，因为客户端和服务器的时间不一定会都是相同的，如果时间不同就会导致问题。所以这个响应头是没有Cache-Control：max-age=*这个响应头准确的，因为max-age=date中的date是个相对时间，不仅更好理解，也更准确。

  4. Last-Modified

    Last-Modified: Dec, 26 Dec 2015 17:30:00 GMT 所请求的对象的最后修改日期(按照 RFC 7231 中定义的“超文本传输协议日期”格式来表示)

  5. Etag

    ETag: "737060cd8c284d8af7ad3082f209582d" 就是一个对象（比如URL）的标志值，就一个对象而言，比如一个html文件，如果被修改了，其Etag也会别修改，所以，ETag的作用跟Last-Modified的作用差不多，主要供WEB服务器判断一个对象是否改变了。比如前一次请求某个html文件时，获得了其 ETag，当这次又请求这个文件时，浏览器就会把先前获得ETag值发送给WEB服务器，然后WEB服务器会把这个ETag跟该文件的当前ETag进行对比，然后就知道这个文件有没有改变了。

  6. Access-Control-Allow-Origin
    • Access-Control-Allow-Origin: *号代表所有网站可以跨域资源共享，如果当前字段为那么Access-Control-Allow-Credentials就不能为true
    • Access-Control-Allow-Origin: www.baidu.com 指定哪些网站可以跨域资源共享

  4. 如果只是获取了dom节点，消耗资源吗

    消耗资源，因为DOM是一个独立于语言的，用于操作XML和HTML文档的程序接口。与JavaScript是相互独立的。每次访问的时候访问的其实是HTML集合，HTML集合是包含DOM节点引用的类数组对象，并不真正包含DOM节点，每次需要新的信息，就需要通过DOM接口查询DOM结构，再返回新的HTML集合。

    1. 需要声明（DOCTYPE），用以说明正在使用的XHTML或者HTML是什么版本。
    2. 需要定义语言编码格式，例如UTF-8
    3. JavaScript定义
    4. css定义

  8. 伪类伪元素区别

    • 借助Promise.rase，只要Promise数组中其他的promise决议了，其他的promise就会被取消

浏览器层合成与页面渲染优化

浏览器渲染流程

对于浏览器如何将一个网页渲染到页面上的，一般可以分为构建DOM树、构建CSSOM渲染树、布局、绘制、渲染层合成这几个步骤

  • 构建DOM树：浏览器将html解析成一个树形结构的DOM树，一般来说，这个过程发生在页面初次加载，或者页面JavaScript修改了节点结构的时候。
  • 构建CSSOM渲染树：浏览器将CSS解析成树形结构的CSSOM树，再和DOM树合成渲染树。
  • 布局（Layout）：浏览器将根据渲染树各个节点的位置和元素信息进行相对位置的计算。一般发生在页面元素几何信息发生改变的时候，也被称为回流。
  • 绘制（Paint）：遍历渲染树，调用渲染期的paint方法在屏幕上绘制出节点内容，本质上是一个像素填充的过程。一般发生在页面初始化和一些不影响布局的css改变中，也被称为重绘。
  • 渲染层合成（Composite）：多个绘制完成后的渲染层按照恰当的重叠顺序进行合并，最后生成位图。

渲染层合成

  • 什么是渲染层合成

    由于在浏览器中，页面的堆叠是依靠类似ps一样的一层层堆叠起来达到层次的效果。在DOM树中每个节点都会对应一个渲染对象，每一层（z-index相同）都是一层渲染层。为了保证页面元素以正确的顺序堆叠，就会出现渲染层合成。

  • 浏览器的渲染原理

    从浏览器的DOM树转化为一个个渲染层，最终合并的过程中，出现了渲染对象、渲染层、图形层、合成层的过渡结构

    1. 渲染对象

      一个 DOM 节点对应了一个渲染对象，渲染对象依然维持着 DOM 树的树形结构。一个渲染对象知道如何绘制一个 DOM 节点的内容，它通过向一个绘图上下文（GraphicsContext）发出必要的绘制调用来绘制 DOM 节点。

    2. 渲染层

      这是浏览器渲染期间构建的第一个层模型，处于相同坐标空间（z轴空间）的渲染对象，都将归并到同一个渲染层中，因此根据层叠上下文，不同坐标空间的的渲染对象将形成多个渲染层，以体现它们的层叠关系。所以，对于满足形成层叠上下文条件的渲染对象，浏览器会自动为其创建新的渲染层。

    3. 图形层

      GraphicsLayer 其实是一个负责生成最终准备呈现的内容图形的层模型，它拥有一个图形上下文（GraphicsContext），GraphicsContext 会负责输出该层的位图。存储在共享内存中的位图将作为纹理上传到 GPU，最后由 GPU 将多个位图进行合成，然后绘制到屏幕上，此时，我们的页面也就展现到了屏幕上。

    4. 合成层

      满足某些特殊条件的渲染层，会被浏览器自动提升为合成层。合成层拥有单独的 GraphicsLayer，而其他不是合成层的渲染层，则和其第一个拥有 GraphicsLayer 的父层共用一个。

      3D transforms：translate3d、translateZ 等
      video、canvas、iframe 等元素
      通过 Element.animate() 实现的 opacity 动画转换
      通过 СSS 动画实现的 opacity 动画转换
      position: fixed
      具有 will-change 属性
      对 opacity、transform、fliter、backdropfilter 应用了 animation 或者 transition
