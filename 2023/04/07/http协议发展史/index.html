http协议发展史

我们知道HTTP是浏览器中最重要且使用最多的协议，它不仅是浏览器与服务端的通信语言，更是互联网的基石。随着浏览器的不断更新迭代，HTTP为了适应技术的更新也在不断进化，学习HTTP的最佳途径就是从浏览器的发展视角来了解HTTP的演进。

Http0.9

Http0.9是一个初期基本协议，只有get请求，只用于获取html

流程：

  • 客户端和服务端建立TCP链接
  • 客户端发起get请求到服务端，请求index.html数据
  • 服务端发回一个Ascii码编码的html字符串

特点：

简单，一个请求一个链接

Http1.0

相比于Http0.9，有以下更新

  • 增加了HEAD、POST等方法
  • 增加了响应状态码，标记可能的错误原因
  • 引入了协议版本号概念（请求行，在http包中第一行，携带版本信息，请求方法等）
  • 引入了HTTP header概念（请求头，使得请求更加灵活）
  • 传输的数据不再局限与文本内容

特点：

请求更加丰富，具有结构性，但是仍然不能进行长连接

Http1.1

相比于Http1.0，有以下更新

  • 长连接：引入TCP连接复用，即一个TCP可以被多个http请求复用
  • 并发连接：对于一个域名的请求允许分配多个长连接（缓解线头阻塞的问题）
  • 增加了PUT、DELETE、OPTIONS、PATCH等方法
  • 新增了缓存相关字段（if-Modified-Since，If-None-Match）
  • 请求头中引入了range字段，支持断点续传
  • 允许响应数据分块（chunked），利于传输大文件
  • 强制要求Host头、让互联网主机托管称为可能
  • 引入安全机制（HTTPS，cookie）

特点：

非常既有革命性的更新，使得http协议完全可广泛使用，但仍有缺点，例如压缩算法不健全，协议包太大，线头阻塞没有完全解决等

Http2.0

相比于Http1.1，有以下更新

  • 支持二进制传输：Http1.1传输的文本内容可以是二进制的，Http2.0中包括头部信息和内容都可以是二进制的，统称为帧
  • HPACK算法压缩头部：使用专用算法压缩头部，减少数据传输量，主要是通过服务端和客户端同时维护一张头部信息表，所有的头部信息在表里面都会有对应的记录，并且会有一个索引号，这样后面只需要发送索引号即可
  • 多路复用： 废弃了 HTTP/1.1 中的管道，同一个TCP连接里面，客户端和服务器可以同时发送多个请求和多个响应，并且不用按照顺序来。由于服务器不用按顺序来处理响应，所以避免了“对头堵塞”的问题。
  • 服务端主动推送：允许服务器主动向客户推送数据
  • 数据流： 由于HTTP/2版本的数据包不是按照顺序发送的，同一个TCP连接里面相连的两个数据包可能是属于不同的响应，因此，必须要有一种方法来区分每一个数据包属于哪个响应。HTTP/2版本中，每个请求或者响应的所有数据包，称为一个数据流（stream），并且每一个数据流都有一个唯一的编号ID，请求数据流的编号ID为奇数，响应数据流的编号ID为偶数。每个数据包在发送的时候带上对应数据流的编号ID，这样服务器和客户端就能分区是属于哪一个数据流。最后，客户端还能指定数据流的优先级，优先级越高，服务器会越快做出响应。

HPACK实现原理：两张表，动态表和静态表

  1. 静态表是将头部的信息如请求协议get用1表示，post用2表示，对照静态表将原本的字符串压缩成数字。
  2. 动态表是将剩下由用户定义的变量用哈夫曼编码优化

Http3.0

添加基于UDP协议的QUIC协议。Http2.0由于采用二进制分帧进行多路复用，通常只使用一个 TCP 连接进行传输，在丢包或网络中断的情况下后面的所有数据都被阻塞。

1.线头阻塞问题：TCP协议重如果又数据丢包，需要等待重传，浪费后续资源

原理：QUIC协议在遇到某个stream传输失败的时候不会阻塞等待，而是继续向下接收数据，因为有一个计数变量，所以哪怕到达顺序不一样也能还原，因为没有线头阻塞，这在文件流传输等不要求高可靠性的场景下很有用。

2.切换网络时候保持连接：从4g切换为5g时候，TCP协议会重连，但是UDP协议通过内建协议不必重连。

原理：QUIC建立协议的时候会随机产生一个64位的ID，接收方确认不依靠ip地址，而是依靠这个ID，所以哪怕网络环境变化，只要ID没变，链接就可以保持。
